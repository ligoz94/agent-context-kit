# Triage Issue Prompt

Use this prompt when a GitHub issue has just been received and needs to be understood, clarified, and broken into developer tasks.

This is **Phase 0** of the pipeline. It runs before `/update-docs` or any implementation.

## When to Use This Prompt

| Situation                                        | Use                                                                   |
| ------------------------------------------------ | --------------------------------------------------------------------- |
| New issue received, no `[DEV]` tasks created yet | **this prompt**                                                       |
| Issue is vague or missing detail                 | **this prompt** — ask questions before anything                       |
| Issue has `[DEV]` tasks but no spec              | [update-docs.md](update-docs.md) or [fix-bug.md](fix-bug.md) directly |
| Spec is approved, ready to code                  | [implement-feature.md](implement-feature.md)                          |

## Required Reading

1. **[Intent Engineering Standard](../values.md)** — agent principles, halt thresholds (READ FIRST)
2. **App's `app-config.md`** — paths, spec system (path in app-config.md § Paths)
3. **App's `docs/README.md`** — feature register, to check if this touches an existing feature

> **MCP Toolshed**: If available, use `list_registry` / `get_feature_doc` instead of manually searching docs.

---

## Process

### Step 1 — Fetch the Issue

```sh
REPO=$(gh repo view --json nameWithOwner --jq .nameWithOwner)
gh issue view <number> --repo "$REPO" --json number,title,body,labels,comments,milestone
```

If no issue number is provided, ask: "Which issue number should I triage?"

Capture:

- Title + body (the stated requirement)
- Labels (bug / feature / enhancement / chore)
- Comments (additional context, decisions, constraints already discussed)
- Milestone (urgency/release signal)

### Step 2 — Check Existing Docs

Search for an existing feature doc or spec that relates to this issue:

```sh
# Search feature docs directory (path from app-config.md § Paths)
grep -ri "<keyword from issue title>" <feature-docs-path>/
```

Also check the app's feature register (`docs/README.md`) for a matching feature row.

Note:

- **Feature doc exists** → this issue is an addition or change to a known feature
- **No feature doc** → this may be a new feature or a standalone fix
- **Spec exists** → implementation may only need a minor spec update or none at all

### Step 3 — Extract Intent

Fill every field of the Intent Extraction Template from [values.md](../values.md#intent-extraction-template-internal-agent-step):

```
Objective:         [What is the goal?]
Constraints:       [What limits exist?]
Non-goals:         [What should NOT be done?]
Data inputs:       [What data comes in?]
Data outputs:      [What data goes out?]
Failure states:    [How can this fail?]
Security:          [Auth, data access, trust boundaries?]
Acceptance criteria: [How do we know it's done?]
```

**If any field cannot be filled from the issue content** → post a comment on the issue with the questions and halt:

```sh
gh issue comment <number> --repo "$REPO" --body "## Triage Questions — needs clarification before decomposition

Before we can break this into developer tasks, we need answers to the following:

**1. [Question about vague objective]**
[Context / why this matters for decomposition]

**2. [Question about missing constraint]**
[Context]

**3. [Question about undefined acceptance criteria]**
[Context]

Once these are answered we can proceed with task decomposition.

---
*Generated by \`/triage-issue\`*"
```

Do **not** infer missing intent. Do **not** create tasks based on assumptions.

### Step 4 — Determine Spec Requirement

Classify the issue to decide whether a spec is needed:

| Signal                                                    | Spec needed?           |
| --------------------------------------------------------- | ---------------------- |
| Bug fix with a clear, bounded root cause                  | No                     |
| Typo, copy change, UI tweak, config value                 | No                     |
| Single-component change, no design decision required      | No                     |
| Spec exists and this only updates one field/section       | No (minor update only) |
| New feature, new user-facing behavior                     | **Yes**                |
| Significant behavior change affecting multiple components | **Yes**                |
| Data model change (new table, column, relation)           | **Yes**                |
| Security boundary change (new auth, new role, new policy) | **Yes**                |
| Anything requiring a design decision before coding        | **Yes**                |

**If unsure**: default to **no spec** for clearly bounded fixes, **yes** for anything with a design decision.

Record the outcome: `Spec needed: yes | no | minor update (<what section>)`

### Step 5 — Decompose into Sub-Issues

Break the fully-clarified intent into the smallest independently-implementable tasks.

#### 5a — Choose task type prefix

| Prefix   | Meaning                                      | Next prompt                                            |
| -------- | -------------------------------------------- | ------------------------------------------------------ |
| `[SPEC]` | Write or update a spec                       | `/update-docs #<number>`                               |
| `[DEV]`  | Implementation (frontend/backend/data model) | `/implement-feature #<number>` or `/fix-bug #<number>` |
| `[TEST]` | Specific test scenarios to verify            | `/ui-test #<number>`                                   |

If a task requires a spec first, create the `[SPEC]` task first and note the ordering dependency.

#### 5b — Find codebase references

Before writing the sub-issue body, search the codebase for relevant files:

```sh
# Find similar feature for pattern reference
grep -r '<keyword>' src/ --include='*.ts' --include='*.tsx' -l | head -10
```

Include the results in the sub-issue body so the implementer knows exactly where to look.

#### 5c — Create sub-issues

Common breakdown axes:
- Backend (API endpoint, service, repository)
- Frontend (UI component, route, state)
- Data model (schema, migration)
- Tests (unit, integration, E2E)
- Docs (spec update, feature doc update)

For each task, create a GitHub sub-issue:

```sh
REPO=$(gh repo view --json nameWithOwner --jq .nameWithOwner)

gh issue create \
  --repo "$REPO" \
  --title "[DEV|TEST|SPEC] <specific task description>" \
  --body "$(cat <<'EOF'
Parent: #<issue-number>

## Context
<1-2 sentences of context from the parent issue>

## Codebase References
- **Pattern to follow:** `<path to similar existing feature>`
- **Files to modify:** `<specific files identified in 5b>`

## Scope
<what this task covers — be explicit about what it does NOT include>

## Acceptance Criteria
<testable criteria for this specific task>

## Next Prompt
\`/implement-feature #THIS\` | \`/fix-bug #THIS\` | \`/update-docs #THIS\` | \`/ui-test #THIS\`
EOF
)"
```

**Rules:**

- No task for vague requirements — resolve Step 3 first
- Each task must be implementable independently (or note explicit ordering)
- `[SPEC]` tasks come before `[DEV]` tasks when a spec is needed
- Always include codebase references — grep before creating the issue

### Step 6 — Hand Off

Output a clear next-step instruction based on Step 4:

**If spec needed:**

> Spec required before implementation. Next: run `/update-docs #<number>` to create the spec from this issue.

**If no spec needed (bug or small fix):**

> No spec required. Next: run `/fix-bug #<number>` or pick up a `[DEV]` sub-issue directly.

**If minor spec update only:**

> Spec exists but needs a minor update. Next: run `/update-docs #<number>` to patch the `<section>` section, then implement.

---

## Anti-Patterns

- **Creating tasks before intent is clear** — gaps in Step 3 must be resolved first
- **Inferring missing requirements** — if the issue doesn't say it, ask, don't guess
- **Always requiring a spec** — small bugs and minor fixes do not need one
- **Never requiring a spec** — features with design decisions always do
- **Over-decomposing** — don't create 10 tasks where 3 would do; match granularity to implementation risk
- **Under-decomposing** — a single `[DEV]` task covering frontend + backend + DB + tests is too large; split it

## Output Format

After completing all steps, summarise:

```
## Triage: #<number> <title>

**Intent**: <one sentence>
**Spec needed**: yes | no | minor update (<section>)
**Tasks created**:
- `[SPEC]` #N1 — <description>
- `[DEV]` #N2, #N3 — <descriptions>
- `[TEST]` #N4 — <description>

**Next step**: <which prompt to run first and on which issue>
```
